# 現行挙動優先ルール（文書だけでは判断できない箇所への対応）

この文書は、`docs/specs` の記述だけでは判断できない箇所で「何を優先するか」を固定する。

## BP-001 Perl参照実装優先
- 仕様と実装がずれた場合、まず Perl 実装の実挙動を優先する。

## BP-002 互換性優先（表示・保存）
- `tree/tree_cat`, `lf/sr`, `resume` は Perl と同等出力を優先する。

## BP-003 legacy構造優先
- legacy 形式（例: 7要素ノード）が混在する場合、Perlが実際に行う走査・再帰条件を優先する。

## BP-004 既知バグ互換
- Perl 側に既知バグ相当の挙動がある場合、研究データ再現性のため同挙動を優先する。

## BP-005 API入力の処理順序
- `rule_name/rule_number` は整合性チェックを行い、不整合はエラーとする。
- single規則の `left` は `check` の alias として扱う。

## BP-006 エラーポリシー
- 不正入力は「黙って補正」せず、`ValueError` / HTTP `400` で明示的に返す。

## BP-007 回帰固定の優先
- 判断が分かれやすい箇所は differential / golden / characterization のテストで固定する。

## BP-008 文書とテストの同期
- ルール変更時は `docs/specs` 配下の文書と契約テストを同時更新する。

## BP-009 報告の仕方
- 報告する際は、必ずスキルを使って環境とブラウザを起動して、ユーザーが現状を確認できるようにする。スキルを使った安定起動に失敗した場合は原因を特定してスキルを改善する。また、git add/commitによりgitレポジトリに現状を記録し、どの報告のタイミングの状態にも戻せるようにする。
